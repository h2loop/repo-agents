## Title: Fix stale protocol state leak in NGAP NG Setup Response handler

### Summary
The NGAP gNB handler for NG Setup Response messages fails to clear existing protocol state before processing new configuration data. This causes memory leaks and stale data accumulation in the `served_guami` and `plmn_supports` linked lists when receiving subsequent NG Setup Response messages. Over time, this leads to memory exhaustion and undefined behavior from processing outdated AMF configuration.

This fix adds comprehensive cleanup logic at the start of `ngap_gNB_handle_ng_setup_response()` that properly frees all existing state in the AMF descriptor, including nested sub-lists for PLMNs, region IDs, AMF set IDs, AMF pointers, and slice support information.

### Changes
- `openair3/NGAP/ngap_gNB_handlers.c`: Added state cleanup routine before NG Setup Response processing. The cleanup uses `STAILQ_FOREACH_SAFE` to safely iterate and free all entries in `served_guami` and `plmn_supports` lists, including their respective sub-lists, preventing memory leaks and stale data issues.

### Implementation Details
The cleanup uses BSD queue macros (`STAILQ_FOREACH_SAFE`) for safe removal during iteration. Each nested data structure is explicitly freed: `served_plmns`, `served_region_ids`, `amf_set_ids`, `amf_pointers` within each GUAMI item, and `slice_supports` within each PLMN support entry. This ensures complete memory deallocation before populating fresh data from the incoming NG Setup Response message.
