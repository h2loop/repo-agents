## Title: Fix use-after-free vulnerability in pnf_sim main()

### Summary
Fixes a use-after-free vulnerability in the pnf_sim main() function where a stack-allocated `pnf_info` structure was passed as user data to `nfapi_pnf_start()`. The stack-allocated object could be accessed after the function returned or after the stack frame was destroyed, leading to undefined behavior and potential crashes.

The root cause was that `pnf_info pnf` was declared as a local stack variable but its pointer (`&pnf`) was stored in the persistent `config->user_data` field. Since `nfapi_pnf_start()` may spawn asynchronous operations or return after the stack frame becomes invalid, any subsequent access to this memory would be unsafe.

The fix converts the `pnf_info` object to heap allocation with proper lifecycle management. The object is now allocated with `new`, passed as a pointer throughout the function, and explicitly deleted after `nfapi_pnf_start()` returns. Additionally, the `nfapi_pnf_config` object is properly destroyed, and error paths now include appropriate cleanup to prevent memory leaks.

### Changes
- `nfapi/open-nFAPI/pnf_sim/src/main.cpp`: 
  - Changed `pnf_info pnf` stack allocation to `pnf_info* pnf = new pnf_info()` heap allocation
  - Updated `read_pnf_xml()` call to dereference pointer: `read_pnf_xml(*pnf, ...)`
  - Set `config->user_data = pnf` instead of `&pnf`
  - Added cleanup after `nfapi_pnf_start()` returns: `delete pnf` and `nfapi_pnf_config_destory(config)`
  - Added `delete pnf` in error path before early return

### Testing
- Verified no regressions in pnf_sim startup and XML configuration parsing
- Confirmed proper memory cleanup using Valgrind memory checker
- Tested error paths to ensure no memory leaks when XML parsing fails
