## Title: Fix race condition in NR UE RRC with mutex protection

### Summary
The NR UE RRC module had a race condition where the `NR_UE_RRC_INST_t` instance was accessed from multiple threads without synchronization, particularly in the `nr_rrc_handle_msg3_indication` code path. This could lead to data corruption when timer state, random access state, and other RRC instance fields were concurrently read and modified.

This fix introduces a per-instance mutex to protect critical sections accessing shared RRC state. The mutex is acquired before accessing or modifying any shared fields in the main RRC message handling functions and released before returning.

### Changes
- `openair2/RRC/NR_UE/rrc_defs.h`: Added `pthread_mutex_t mutex` field to `NR_UE_RRC_INST_t` structure for thread-safe access to shared RRC instance data.
- `openair2/RRC/NR_UE/rrc_UE.c`: 
  - Initialize mutex in `nr_rrc_init_ue()` during RRC instance creation
  - Added `pthread_mutex_lock/unlock` pairs in `nr_rrc_handle_msg3_indication()` to protect RA trigger handling and timer access
  - Added mutex protection in `nr_rrc_handle_ra_indication()` for RA success handling
- `openair2/RRC/NR_UE/rrc_timers_and_constants.c`: Added mutex lock in `nr_rrc_handle_timers()` to protect timer tick updates and state transitions

### Implementation Details
The mutex is scoped per RRC instance (`NR_UE_RRC_INST_t`) and follows a simple lock/unlock pattern around critical sections. The implementation avoids nested locks and ensures the mutex is always unlocked before exiting the protected region, even on error paths. This minimal approach adds synchronization only where shared state is actually accessed, minimizing performance impact while eliminating the race condition.
