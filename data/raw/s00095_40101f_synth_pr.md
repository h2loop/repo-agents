## Title: Fix buffer overflow vulnerabilities in eNB thread name formatting

### Summary
Replace unsafe `sprintf()` calls with bounded `snprintf()` when formatting thread names in the eNB L1 processing threads. The original code used unbounded string formatting that could potentially overflow the 100-byte thread name buffers, and included unnecessary newline characters in the thread names.

### Changes
- `executables/lte-enb.c`: 
  - In `L1_thread_tx()`: Replace `sprintf()` with `snprintf(thread_name, sizeof(thread_name), ...)` and remove trailing `\n` from format string
  - In `L1_thread()`: Replace `sprintf()` with `snprintf(thread_name, sizeof(thread_name), ...)` and remove trailing `\n` from format string
  - Extract thread ID ternary expression to separate variable for improved readability

### Implementation Details
The thread name buffers are stack-allocated as `char thread_name[100]`. While the formatted names ("TXnp4_0" or "RXn_TXnp4_1") are well under this limit, using `snprintf()` eliminates any risk of buffer overflow and follows secure coding practices. The newline characters were artifacts that caused thread names to include unexpected whitespace.

### Testing
- Code compiles without warnings with `-Wformat-security` flags
- Thread names are correctly formatted and displayed in logs without embedded newlines
- Static analysis tools no longer flag these locations as buffer overflow risks
