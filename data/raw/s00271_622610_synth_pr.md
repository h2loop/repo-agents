## Title: Fix LDPC queue priority inversion in DPDK bbdev configuration

### Summary
Correct a scheduling priority inversion in the DPDK baseband device (bbdev) queue configuration where LDPC encoding operations were incorrectly assigned higher priority than decoding operations. In 5G systems, LDPC decoding latency is critical for downlink throughput and should be prioritized over encoding. The original code configured encoding queues first with priority 0 (highest) and decoding queues second with priority 1, causing lower-priority encoding traffic to preempt decoding operations.

This fix reverses the configuration order: decoding queues are now configured first with priority 0, and encoding queues are configured second with priority 1. The queue ID assignments in the `active_dev` structure are correspondingly swapped to maintain correct operation. No functional changes to the encoding/decoding logic itself.

### Changes
- `openair1/PHY/CODING/nrLDPC_coding/nrLDPC_coding_aal/nrLDPC_coding_aal.c`: 
  - Swapped the order of queue configuration in `add_dev()` function (lines 397-420)
  - LDPC decoding queues now configured first with higher priority (priority 0)
  - LDPC encoding queues now configured second with lower priority (priority 1)
  - Updated `active_dev.dec_queue` and `active_dev.enc_queue` assignments accordingly
  - Corrected log messages to reflect actual operation type and priority

### Testing
- Verified DPDK bbdev initialization logs show decoding queues at priority 0 and encoding queues at priority 1
- Confirmed no regression in LDPC encoding/decoding functionality through basic NR PHY validation
- Queue priority ordering now aligns with 5G system requirements where decoding latency is performance-critical
