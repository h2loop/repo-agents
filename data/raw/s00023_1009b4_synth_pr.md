## Title: Fix use-after-free vulnerability in AT+COPS test command parser

### Summary
The `parse_cops_test` function in the NAS UE AT command parser did not initialize the `mask` field when handling the AT+COPS=? test command. This omission created a use-after-free vulnerability, as downstream command processing logic relies on the mask to determine whether parameter fields contain valid allocated memory that requires cleanup. Without the `AT_COMMAND_NO_PARAM` flag set, the parser could incorrectly attempt to free uninitialized memory, leading to undefined behavior and potential security issues.

The fix explicitly sets `at_command->mask = AT_COMMAND_NO_PARAM` to clearly indicate that this command variant has no parameters, preventing any subsequent processing steps from misinterpreting the command structure.

### Changes
- `openair3/NAS/UE/API/USER/at_command.c`: Added command mask initialization in `parse_cops_test()` to properly flag the AT+COPS test command as having no parameters.

### Implementation Details
The `mask` field is a critical part of the `at_command_t` structure that controls memory management and validation logic throughout the AT command processing pipeline. For test commands like AT+COPS=? that carry no operator parameters, setting `AT_COMMAND_NO_PARAM` ensures that:
- Memory cleanup functions skip uninitialized parameter fields
- Validation logic doesn't attempt to access non-existent parameter data
- The command state machine correctly handles the no-parameter case

This is a minimal, targeted fix that addresses the root cause without affecting other command parsing logic.

### Testing
- Code review confirms the mask field is properly checked in command processing and cleanup paths
- Static analysis verifies no other code paths attempt to access parameter fields when `AT_COMMAND_NO_PARAM` is set
