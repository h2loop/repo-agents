## Title: Fix memory leak in F1AP ID management unit tests

### Summary
The unit test functions `cu_data_test()` and `du_data_test()` allocate hashtables via `cu_init_f1_ue_data()` and `du_init_f1_ue_data()` but never free them, causing memory leaks on both success and error paths. This is particularly problematic when running tests under valgrind or in continuous integration environments where tests execute repeatedly.

I add symmetric cleanup functions to properly destroy the hashtables and update the tests to invoke them before returning. The cleanup functions include proper mutex handling and NULL pointer checks to ensure safe, idempotent operation.

### Changes
- **openair2/F1AP/f1ap_ids.c**: Add `cu_destroy_f1_ue_data()` and `du_destroy_f1_ue_data()` functions. Each locks its respective mutex, checks if the hashtable pointer is non-NULL, calls `hashtable_destroy()` to free all entries and the table itself, sets the pointer to NULL, and unlocks the mutex.
- **openair2/F1AP/f1ap_ids.h**: Add declarations for `cu_destroy_f1_ue_data()` and `du_destroy_f1_ue_data()` to the public API.
- **openair2/F1AP/f1ap_ids_test.c**: Update `cu_data_test()` and `du_data_test()` to call their respective cleanup functions at function exit, ensuring resources are freed regardless of test outcome.

### Implementation Details
The destroy functions follow the same mutex pattern as the init functions for thread safety. The NULL check after destruction prevents double-free issues and use-after-free bugs if the functions are called multiple times. No functional behavior changes; this is purely a resource management fix.

### Testing
- Verified no memory leaks with `valgrind --leak-check=full` on `f1ap_ids_test`
- All existing assertions in the test functions continue to pass
- Cleanup functions safely handle multiple invocations without crashing
