## Title: Fix incorrect boolean literal passed as flag to put_UE_in_freelist

### Summary
The `rx_sdu` function in the LTE MAC UL scheduler incorrectly passes a boolean `true` as the third parameter to `put_UE_in_freelist()`, but the function expects a numeric flag value. This occurs in the Random Access failure handling path when Msg3 HARQ retransmissions exceed the configured maximum. While `true` typically evaluates to `1`, using a boolean literal for a flag parameter is semantically incorrect and fragile against flag definition changes or compiler differences. This fix ensures the correct explicit flag value is passed.

### Changes
- `openair2/LAYER2/MAC/eNB_scheduler_ulsch.c`: Changed the third parameter in `put_UE_in_freelist()` call from `true` to `1` in the RA failure handling path (line 335). This correctly passes the explicit flag value expected by the function for RNTI release operations.

### Testing
- Verified RA procedure cleanup behavior in simulation scenarios where Msg3 HARQ fails after maximum retransmissions
- Confirmed no functional regression in RNTI management and RA state cleanup
