## Title: Fix race condition in NFAPI pusharrays32 serialization

### Summary
The `pusharrays32()` function in the NFAPI common library performed unsynchronized access to shared data structures when serializing int32_t arrays, leading to race conditions in multi-threaded environments. Multiple threads calling this function concurrently could corrupt the output buffer pointer or intermediate state, causing data corruption, buffer overruns, or intermittent crashes during nFAPI message encoding.

The root cause was the lack of synchronization primitives protecting the critical section that modifies the output buffer pointer and validates array bounds. This manifested particularly under high load when multiple L1/L2 interfaces processed slot-indications and config requests simultaneously.

The fix introduces a static mutex to serialize access to the critical section, ensuring thread-safe array serialization. All exit paths properly release the mutex, and error handling is added for mutex acquisition failures.

### Changes
- **nfapi/open-nFAPI/common/src/nfapi.c**: 
  - Added static mutex `pusharrays32_mutex` with `PTHREAD_MUTEX_INITIALIZER`
  - Wrapped function logic with `pthread_mutex_lock()` at entry and `pthread_mutex_unlock()` before all return paths
  - Added error trace logging if mutex acquisition fails
  - Ensured mutex is released on both success and error conditions (array bounds violation, buffer space exhaustion, and pushs32 failures)

### Implementation Details
- Mutex scope is limited to this translation unit to minimize contention
- Lock acquisition is the first operation after function entry to protect all shared state
- Early return paths for `len == 0` and bounds violations properly unlock before returning
- The mutex unlock precedes `on_error()` calls to avoid holding locks during error handling

### Testing
- Validated with TSAN (ThreadSanitizer) to confirm no remaining data races in nFAPI serialization paths
- Ran multi-threaded unit tests simulating concurrent config and slot indication messages
- Verified no performance regression in single-threaded operation through nfapi message encoding benchmarks
- Tested error paths to ensure mutex is always released even under failure conditions
