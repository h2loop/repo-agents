## Title: Fix CRC errors in ESM timer initialization due to message length field ordering

### Summary
The ESM (EPS Session Management) timer initialization in the NAS UE layer contained a critical initialization order bug that caused downstream CRC computation errors. Specifically, in `esm_pt_start_timer()`, the `ctx->args->msg.length` field was being set to 0 *after* the message buffer allocation, but before the actual length was assigned. If memory allocation failed, the length field could remain uninitialized, causing CRC verification logic to read garbage memory and produce false positives or missed error detection. Additionally, the function lacked proper error handling for malloc failures, potentially leaving the system in an inconsistent state.

This fix corrects the initialization order by setting `msg.length` to 0 immediately after allocating the args structure, ensuring consistent state regardless of allocation success. It also adds proper error handling with explicit checks for both the args structure and message buffer allocations, with appropriate error logging and cleanup.

### Changes
- `openair3/NAS/UE/ESM/esm_pt.c`: 
  - Moved `ctx->args->msg.length = 0` initialization before the message buffer malloc call to ensure proper field initialization
  - Added error handling for `ctx->args->msg.value` malloc failure with cleanup and error logging
  - Added error handling for `ctx->args` malloc failure with error logging
  - Ensures consistent state management and prevents downstream CRC errors

### Testing
- Verified NAS ESM procedure retransmission behavior remains correct for successful allocations
- Confirmed proper error path execution when memory allocation fails
- Validated that downstream CRC computation no longer reads uninitialized memory lengths
- Tested with various ESM message types to ensure no regression in timer-based retransmissions
