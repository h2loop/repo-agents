## Title: Fix DRB ID range validation to prevent resource leaks and configuration errors

### Summary
The PDCP UE manager used a hardcoded limit of 5 for DRB ID validation and cleanup operations, while the system architecture supports MAX_DRBS_PER_UE (32) data radio bearers. This inconsistency caused two critical issues:

1. **Configuration rejection**: The validation in `nr_pdcp_ue_add_drb_pdcp_entity()` rejected valid DRB IDs between 6-32, artificially limiting UE capacity to 5 DRBs despite system support for 32.

2. **Resource leaks**: The cleanup loop in `nr_pdcp_manager_remove_ue()` only iterated through the first 5 DRBs, leaving entities 6-32 allocated when removing UE contexts, causing memory leaks and resource exhaustion over time.

This fix replaces the hardcoded value of 5 with the MAX_DRBS_PER_UE constant throughout the PDCP UE manager, ensuring consistent DRB ID range handling across validation and lifecycle management.

### Changes
- `openair2/LAYER2/nr_pdcp/nr_pdcp_ue_manager.c`:
  - `nr_pdcp_ue_add_drb_pdcp_entity()`: Updated DRB ID validation to check `drb_id > MAX_DRBS_PER_UE` instead of `drb_id > 5`
  - `nr_pdcp_manager_remove_ue()`: Expanded cleanup loop from `j < 5` to `j < MAX_DRBS_PER_UE`

### Testing
- Verified acceptance of DRB IDs across full valid range (1-32)
- Confirmed complete cleanup of all DRB entities during UE context removal
- Validated no regression in PDCP data path functionality
