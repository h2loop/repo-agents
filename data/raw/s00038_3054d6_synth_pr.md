## Title: Fix uninitialized SIMD accumulators in DFT causing missed UE scheduling

### Summary
The butterfly-3 DFT functions in the PHY signal processing chain contained uninitialized SIMD accumulator registers, corrupting channel estimates and causing the MAC scheduler to miss valid scheduling opportunities for UEs with pending data. The `tmpre` and `tmpim` accumulators in `bfly3_256()` and `bfly3_tw1_256()` were not zeroed before `cmac_256()` operations, leading to stale values being accumulated into the DFT outputs. This propagated corrupted CSI to the scheduler, which then incorrectly assessed UE channel quality and skipped scheduling decisions for UEs that should have been allocated resources.

The bug manifested under specific RF conditions where the butterfly-3 code paths were exercised, particularly with certain PRB allocations and antenna configurations. The uninitialized accumulators caused intermittent but severe throughput degradation that was difficult to reproduce.

### Changes
- **openair1/PHY/TOOLS/oai_dfts.c**: 
  - Initialize `tmpre` and `tmpim` to zero using `simde_mm256_setzero_si256()` before the first `cmac_256()` call in `bfly3_256()` (line 517) and `bfly3_tw1_256()` (line 591)
  - Re-initialize accumulators to zero before the second set of `cmac_256()` operations in both functions (lines 525 and 599) to ensure independent computation paths

### Implementation Details
The fix adds explicit zero-initialization of the 256-bit SIMD accumulators at the correct locations in the butterfly-3 computation flow. This ensures that each `cmac_256()` operation starts from a clean state, preventing garbage accumulation from previous computations or stack memory. The fix is minimal and targeted, preserving the optimized SIMD code structure while correcting the mathematical error.

### Testing
- Verified correct DFT output against a golden reference for all 3GPP-defined DFT sizes
- Ran `nr_ulsim` with 8 UEs and full buffer traffic; confirmed scheduler now correctly allocates resources to all UEs with pending data
- No performance regression observed in PHY processing latency with the added zero-initialization instructions
