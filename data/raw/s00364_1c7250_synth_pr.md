## Title: Replace magic number with symbolic constant for PDU type field size

### Summary
The nFAPI packing functions in `nfapi_p7.c` used the magic number `1` in five locations to represent the size of the PDU type field when calculating PDU sizes. This pattern, repeated with identical comments ("add 1 for the pdu_type"), made the code's intent unclear and maintenance error-prone. This change introduces a symbolic constant `NFAPI_PDU_TYPE_FIELD_SIZE` to explicitly document this field size, improving code readability and maintainability. No functional behavior is changed.

### Changes
- `nfapi/open-nFAPI/nfapi/public_inc/nfapi_interface.h`: Added `NFAPI_PDU_TYPE_FIELD_SIZE` constant (value: 1) to the public interface header for visibility across packing functions
- `nfapi/open-nFAPI/nfapi/src/nfapi_p7.c`: Replaced magic number `1` with `NFAPI_PDU_TYPE_FIELD_SIZE` in five packing functions:
  - `pack_dl_config_request_body_value()`
  - `pack_ul_config_request_body_value()`
  - `pack_hi_dci0_request_body_value()`
  - `pack_lbt_dl_config_request_body_value()`
  - `pack_lbt_dl_config_indication_value()`

### Implementation Details
The constant is defined in the public interface header (`nfapi_interface.h`) alongside other nFAPI length constants like `NFAPI_HEADER_LENGTH`. This centralizes size-related magic numbers and makes the 1-byte PDU type field size self-documenting. The replacement is a direct 1:1 mapping that preserves all existing PDU packing logic and binary output.

### Testing
- Verified compilation succeeds with the new constant
- Confirmed PDU size calculations remain unchanged (constant value is identical to the replaced magic number)
- No runtime behavior changes expected; this is a pure code clarity improvement
