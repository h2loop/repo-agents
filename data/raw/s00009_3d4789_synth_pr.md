## Title: Fix nFAPI timing_window configuration to comply with 3GPP requirements

### Summary
The nFAPI timing_window parameter was incorrectly configured with a value of 30 microseconds instead of the required 30000 microseconds (30ms) as per 3GPP specifications. This misconfiguration caused timing violations downstream of the PDU processing pipeline, potentially leading to L2 messages being incorrectly marked as late and dropped. The root cause was a unit conversion error where the timing window duration was set to an incorrect default value in multiple nFAPI configuration paths.

### Changes
- `nfapi/oai_integration/nfapi_pnf.c`: Correct default timing_window from 30 to 30000 microseconds when nFAPI TLV is not present
- `nfapi/oai_integration/nfapi_vnf.c`: Fix VNF timing_window initialization from 30 to 30000 microseconds
- `nfapi/open-nFAPI/vnf/src/vnf_interface.c`: Update PHY timing_window allocation from 30 to 30000 microseconds
- `nfapi/tests/p5/nr_fapi_config_request_test.c`: Adjust test configuration to use correct 30000 microsecond value

### Implementation Details
The timing_window parameter defines the allowable timing deviation for nFAPI messages between PNF and VNF components. According to 3GPP specifications, the default value must be 30000 microseconds (30ms), not 30 microseconds. This fix ensures consistent behavior across all nFAPI configuration paths and prevents false detection of late messages in the `pack_rx_ulsch_indication_body_value` processing pipeline and related functions.

### Testing
- Verified compliance with 3GPP timing requirements
- Ensures proper message processing without false "late message" detection
- Maintains backward compatibility with existing nFAPI implementations
