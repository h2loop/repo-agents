## Title: Add defensive state validation to config_sib13 MBSFN configuration

### Summary
The `config_sib13` function in the LTE MAC layer previously lacked validation of input parameters and system state before processing MBSFN area configuration. This created a risk of null pointer dereferences and out-of-bounds memory access if the function was called with invalid parameters or before proper system initialization. The missing checks could lead to crashes during eNB startup or runtime reconfiguration, particularly in multi-cell deployments where MBSFN configuration is dynamic.

This patch adds comprehensive defensive checks at the entry point of `config_sib13` to validate all inputs and critical data structures. The validation ensures that module IDs are within valid bounds, component carrier indices are non-negative, and all required MAC layer structures (RC.mac, configuration, and interface instances) are properly initialized before any protocol message processing occurs.

### Changes
- `openair2/LAYER2/MAC/config.c`: Added `AssertFatal` validation checks for `Mod_id`, `CC_id`, and `mbsfn_Area_idx` parameters. Added null pointer checks for `RC.mac`, `RC.mac[Mod_id]`, `RC.mac[Mod_id]->config`, and `RC.mac[Mod_id]->if_inst` before proceeding with MBSFN configuration logic.

### Implementation Details
The validation uses `AssertFatal` macros for immediate failure detection with descriptive error messages indicating the specific validation failure and affected eNB module. The `Mod_id` check validates against `MAX_MAC_INST` to prevent out-of-bounds access to the MAC instances array. All checks are placed at the function entry point, following the pattern of other configuration functions in the codebase. This ensures early failure detection while maintaining consistency with existing error handling conventions in the OAI MAC layer.
