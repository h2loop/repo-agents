## Title: Add state validation for DCI message processing in LTE PHY downlink simulation

### Summary
The LTE PHY downlink simulator processes DCI (Downlink Control Information) messages without validating whether any DCI messages are actually available. When `num_dci` is zero after PDCCH detection, the code continues to process downstream protocol messages, leading to potential undefined behavior from operating on empty DCI allocations. This manifests as either silent failures or crashes when the simulation attempts to generate downlink parameters from non-existent DCI messages.

This fix adds explicit state validation before DCI processing. The code now checks if `num_dci == 0` after PDCCH processing, logs a clear error message, increments the DCI error counter for the current HARQ round, and breaks from the processing loop. This prevents the simulation from proceeding with invalid state and provides proper error visibility.

### Changes
- `openair1/SIMULATION/LTE_PHY/dlsim.c`: Added validation check for `num_dci` before DCI message processing in the main simulation loop. When no DCI messages are available, the code now logs an error and increments `dci_errors[round]` before breaking.

### Implementation Details
The guard clause is inserted immediately after the PDCCH processing stage and before the `memcpy` operation that copies DCI allocations. This ensures we never attempt to copy zero DCI entries or call `generate_ue_dlsch_params_from_dci()` with invalid parameters. The error logging uses the existing PHY logging framework (`LOG_E`) for consistency with other error conditions in the codebase.

### Testing
- Verified that simulation scenarios with failed PDCCH detection now produce clear error messages instead of potential segmentation faults
- Confirmed DCI error counters properly track these events for statistical analysis
- Full downlink simulation test suite passes with no regressions
