## Title: Fix PUSCH timing advance estimation by processing both DMRS symbols

### Summary
The RU uplink processing procedure was incorrectly extracting and processing only a single DMRS symbol for PUSCH channel estimation. This incomplete processing led to inaccurate timing advance calculations and violated the expected 3GPP signaling flow for uplink synchronization. According to 3GPP specifications, both DMRS symbols within a slot should be processed to enable proper channel interpolation and reliable timing detection.

The root cause was the absence of second DMRS symbol processing in `ru_fep_full_2thread()`, which prevented the timing advance estimation algorithm from receiving complete channel state information. This fix adds extraction and channel estimation for the second DMRS symbol (symbol 10), enabling dual-symbol interpolation for robust timing advance calculation.

### Changes
- `openair1/SCHED/ru_procedures.c`: Added processing block for second DMRS symbol including `ulsch_extract_rbs_single()` and `lte_ul_channel_estimation_RRU()` calls. The interpolation flag is enabled to leverage both symbols for improved channel estimation quality before calling `lte_est_timing_advance_pusch()`.

### Implementation Details
The second DMRS symbol is processed with `interpolate=1` to combine information from both symbols, significantly improving channel estimate accuracy. The timing advance check now operates on complete channel information, ensuring proper synchronization state transitions in `RU_CHECK_SYNC` mode.

### Testing
- Verified timing advance estimation accuracy improves with dual-symbol processing
- Confirmed correct state transitions in RU synchronization procedure
- Validated against 3GPP TS 38.211 DMRS positioning requirements
