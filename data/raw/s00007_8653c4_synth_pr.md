## Title: Fix race condition in timeline widget allocator

### Summary
The `allocate` function in the timeline GUI component lacked proper synchronization when modifying shared widget data, creating a race condition under concurrent access. The function directly manipulated widget properties (`common.x`, `common.y`, `common.width`, `common.height`) and performed memory reallocation on `this->s[i].color` without acquiring the GUI lock, while other functions in the same file correctly use `glock()`/`gunlock()` pairs.

This bug could manifest when multiple threads interact with the GUI simultaneously, potentially causing data corruption, inconsistent widget state, or memory access violations during timeline resizing operations. The fix adds the missing lock guards to ensure thread-safe access to shared data structures.

### Changes
- `common/utils/T/tracer/gui/timeline.c`: Added proper synchronization to the `allocate` function:
  - Declared `struct gui *g = _gui;` to access the GUI structure
  - Added `glock(g)` at function entry to acquire the GUI lock before any shared data access
  - Added `gunlock(g)` at function exit to release the lock after all modifications complete
  - This aligns with the synchronization pattern already established in neighboring functions (`paint`, `button`, etc.)

### Implementation Details
The lock scope encompasses all shared state modifications including widget geometry updates and the `realloc` operation on the color buffer array. The `gui_notify()` call remains inside the locked region to ensure consistent state during notification callbacks.

### Testing
- Verified the fix follows the established locking convention used throughout `timeline.c`
- Code review confirms no other code paths bypass the now-protected region
- The change is minimal and surgical, reducing risk of introducing new issues
