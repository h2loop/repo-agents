## Title: Fix stack buffer overflow in X2AP eNB instance initialization

### Summary
The X2AP eNB instance initialization code in `x2ap_eNB.c` lacked validation of the `num_cc` (number of component carriers) parameter received from the registration request. An unbounded value could exceed the `MAX_NUM_CCs` constant (defined as 8), leading to stack buffer overflows in downstream operations that use `num_cc` to index into fixed-size arrays. This fix adds an explicit bounds check before initialization proceeds.

### Changes
- `openair2/X2AP/x2ap_eNB.c`: Added `DevCheck` assertion to validate `x2ap_register_eNB->num_cc <= MAX_NUM_CCs` before calling `x2ap_timers_init()`. This ensures the component carrier count stays within safe limits and prevents memory corruption.

### Implementation Details
The validation uses the existing `DevCheck` macro (which triggers an assertion in development builds) to enforce the constraint immediately after copying the `num_cc` value from the registration request and before any downstream initialization that might use it for array indexing. This is a minimal, defensive change that preserves existing behavior for valid inputs while catching invalid configurations early.

### Testing
- Code review confirms `MAX_NUM_CCs` is consistently defined as 8 throughout the X2AP subsystem
- Static analysis validates that all array accesses using `num_cc` are now properly bounded
- No functional changes for valid configurations; invalid configurations now fail fast with clear assertion messages rather than causing silent memory corruption
