## Title: Fix uninitialized vendor extension in NR Start Request unpacker

### Summary
The `unpack_nr_start_request` function in the nFAPI P5 protocol layer fails to initialize the `vendor_extension` field when unpacking incoming messages. This causes configuration mismatches and inconsistent downstream behavior, as uninitialized memory values may be interpreted as valid TLV tags or lengths, leading to potential parsing errors or segmentation faults.

The root cause is missing initialization of the `vendor_extension` structure members before calling the TLV unpacking routine. This fix initializes the `vendor_extension` tag, length, and value fields to safe defaults (`NFAPI_VENDOR_EXTENSION_TAG` with zero length) before invoking `unpack_nr_tlv_list()`. This ensures deterministic behavior whether or not the vendor extension IE is present in the message, aligning with the nFAPI specification's handling of optional information elements.

### Changes
- `nfapi/open-nFAPI/nfapi/src/nfapi_p5.c`: Added initialization of `vendor_extension.tl.tag`, `vendor_extension.tl.length`, and `vendor_extension.value` fields in `unpack_nr_start_request()` to prevent uninitialized memory access.

### Testing
- Verified consistent behavior in downstream configuration processing
- Confirmed no regressions in existing nFAPI P5 protocol operations
