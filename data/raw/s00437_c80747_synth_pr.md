## Title: Fix integer overflow in UE RX resynchronization loops

### Summary
The UE RX resynchronization logic in `syncInFrame()` uses a 32-bit `int` for the loop counter when iterating over sample offsets. When `rx_offset` (an `openair0_timestamp`, typically 64-bit) exceeds INT_MAX, storing it in a 32-bit integer causes overflow, leading to incorrect loop behavior and potential system instability. This affects both IQ player/recorder modes and normal operation paths. I promote the loop variable type to `int64_t` to match the `rx_offset` parameter, ensuring correct handling of large timestamp values during resynchronization.

### Changes
- `executables/nr-ue.c`: Change loop variable type from `int` to `int64_t` in two resynchronization loops within `syncInFrame()` to prevent integer overflow

### Implementation Details
The fix addresses two separate loops that process `rx_offset` in chunks: one for slot-based resynchronization (IQ player/recorder mode) and one for subframe-based resynchronization. Both now use `int64_t size = rx_offset` to maintain the full precision of the timestamp value throughout the decrement operations.
