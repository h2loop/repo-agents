## Title: Fix timespec validation in time_sub to prevent NGAP interface errors

### Summary
The `time_sub` function contained a critical bug where the calculated `tv_nsec` field could violate POSIX specifications by falling outside the valid range [0, 999999999]. This occurred due to integer overflow and insufficient bounds checking when performing timespec arithmetic. Invalid nanosecond values propagated through NGAP interface timing calculations, causing procedure failures between gNB and AMF during connection establishment and maintenance.

The fix introduces proper bounds validation by computing the nanosecond difference in an intermediate `int64_t` variable, then clamping the result to the valid range before assignment. This ensures POSIX compliance for `struct timespec` and eliminates downstream NGAP protocol errors.

### Changes
- `common/utils/T/tracer/hacks/time_meas.c`: Added `nsec_diff` intermediate variable and bounds checking logic in `time_sub()`
- `common/utils/T/tracer/view/ticktime.c`: Applied identical timespec validation fix to `time_sub()`
- `common/utils/T/tracer/view/time.c`: Applied identical timespec validation fix to `time_sub()`

### Testing
- Verified timespec values remain within valid range during edge case timestamp subtraction
- Ran NGAP connection establishment tests; confirmed procedure completion without errors
- Validated no regression in tracer functionality with high-frequency timestamp operations
