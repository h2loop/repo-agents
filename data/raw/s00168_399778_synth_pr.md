## Title: Fix PUCCH resource configuration arrays causing HARQ process errors

### Summary
The NR MAC gNB scheduler contains configuration arrays for default PUCCH resources indexed by `pucch_ResourceCommon` (valid range 0-15). Two critical issues were identified that caused HARQ process failures and incorrect redundancy version selection during retransmissions:

1. **Array bounds error**: `default_pucch_numbsymb` contained 17 elements instead of 16, creating an off-by-one overflow when indexed by `pucch_ResourceCommon`
2. **Incorrect resource mapping**: `default_pucch_csset` contained wrong values that led to improper PUCCH resource set identification, causing HARQ feedback to use incorrect redundancy versions

These configuration errors manifested as downstream HARQ process failures when `nr_get_default_pucch_res()` was called during UE scheduling. The patch corrects both arrays to align with 3GPP PUCCH resource mapping specifications.

### Changes
- `openair2/LAYER2/NR_MAC_gNB/gNB_scheduler_primitives.c`:
  - Fixed `default_pucch_numbsymb` array size from 17 to 16 elements (removed trailing element)
  - Corrected `default_pucch_csset` values from `{2,3,3,2,4,4,4,2,4,4,4,2,4,4,4,4}` to `{3,3,3,3,4,4,4,3,4,4,4,3,4,4,4,4}` for proper PUCCH resource set mapping

### Implementation Details
The `pucch_ResourceCommon` parameter is validated to range 0-15 via `AssertFatal`, requiring exactly 16-element arrays. The corrected `csset` values ensure proper mapping to PUCCH resource sets, which is critical for HARQ ACK/NACK feedback and redundancy version handling during retransmissions.

### Testing
- Verified array bounds with all valid `pucch_ResourceCommon` values (0-15)
- Confirmed HARQ processes now select correct redundancy versions during retransmissions
- Validated PUCCH resource allocation across different UE configurations
- Ensured no regression in PUCCH format and symbol configuration arrays
