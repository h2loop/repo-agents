## Title: Fix race condition in MAC stats telnet command affecting HARQ process state

### Summary
The `measurcmd_display_macstats_ue` telnet command was reading UE MAC statistics without proper synchronization, causing race conditions with the MAC layer's HARQ process updates. When the telnet server accessed `UE_info` and `eNB_UE_STATS` structures while they were being modified by the scheduler, it could observe inconsistent HARQ state information. This manifested downstream as incorrect retransmission decisions or wrong redundancy version (RV) selections, particularly under high load when telnet queries coincided with HARQ updates.

The fix adds proper mutex protection around the critical section that iterates through UE list and accesses shared MAC statistics structures.

### Changes
- `common/utils/telnetsrv/telnetsrv_enb_measurements.c`: Added `pthread_mutex_lock(&RC.mac[eNB_id]->UL_INFO_mutex)` before the UE iteration loop and `pthread_mutex_unlock()` after completion to ensure atomic access to shared MAC data structures.

### Implementation Details
The `UL_INFO_mutex` is the existing synchronization primitive used by the MAC layer to protect UE information and HARQ state updates. The lock is held for the duration of the entire stats collection loop to maintain a consistent snapshot of all UE statistics. A local pointer `mac` was introduced to improve code readability.

### Testing
- Verified telnet commands no longer trigger HARQ state corruption under concurrent MAC scheduler load
- Confirmed `measurcmd_display_macstats_ue` returns consistent results across multiple invocations
- No performance regression observed in MAC scheduler latency with mutex acquisition
