## Title: Fix double-free vulnerability in NR UE initial synchronization

### Summary
The `nr_initial_sync` function contains a double-free bug when copying synchronization results. When a cell is detected, the code performs a struct copy (`res = *ssbInfo`) that duplicates the `rxdata` pointer array. The original `ssbInfo->rxdata` buffers are subsequently freed within the function, but the copied `res.rxdata` pointers remain dangling. When the caller later frees the `res` structure, the same memory is released again, causing undefined behavior and potential crashes.

This issue manifests during UE cell search and initial synchronization procedures, particularly when the first detected cell is selected for further processing.

### Changes
- `openair1/PHY/NR_UE_TRANSPORT/nr_initial_sync.c`: Added `res.rxdata = NULL;` immediately after the struct assignment to break the pointer aliasing, preventing the double-free while preserving all other synchronization result data.

### Implementation Details
The fix is minimal and surgical, inserting a single null pointer assignment with a clarifying comment. This approach maintains the existing memory management pattern—where `ssbInfo` owns and frees the rxdata buffers—while ensuring the copied result structure doesn't retain stale pointers. The null assignment occurs at line 355, right after `res = *ssbInfo;` within the cell detection block.

### Testing
- Verified memory is freed exactly once per execution path using valgrind
- Confirmed no regression in cell detection success rates
- Checked that synchronization result fields remain intact and usable by callers
- Ran NR UE initial sync test cases for FR1 and FR2 configurations
