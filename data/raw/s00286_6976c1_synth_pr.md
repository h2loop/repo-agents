## Title: Fix missed scheduling opportunity for UEs with pending data

### Summary
The RAN Control (RC) function's event generation thread was not properly scheduling UEs with pending data, causing missed transmission opportunities. The `emulate_rrc_msg()` function was generating random indication data without checking for UEs that have buffered data waiting for transmission. This resulted in UEs with pending RLC data being overlooked during scheduling rounds, increasing latency and risking buffer overflow.

The fix introduces a new `fill_report_style_4()` function that explicitly schedules UEs with pending data before generating RC indication messages. This ensures that UEs requiring transmission resources are prioritized in the scheduling process.

### Changes
- **openair2/E2AP/RAN_FUNCTION/O-RAN/ran_func_rc.c**: 
  - Modified `emulate_rrc_msg()` to call `fill_report_style_4()` instead of directly using `fill_rnd_rc_ind_data()`
  - Added `fill_report_style_4()` function to schedule UEs with pending data and generate indication messages
  - Added `schedule_pending_data_ues()` helper function to query MAC layer for UEs with non-empty RLC buffers and trigger scheduling

### Implementation Details
The new `fill_report_style_4()` function integrates scheduling logic into the RC indication generation process. It first calls `schedule_pending_data_ues()` to ensure any UE with pending data is marked for scheduling before the indication message is constructed. The scheduling function iterates through connected UEs, checks their RLC buffer status, and triggers the MAC scheduler to allocate resources for UEs with data to send. This closes the gap where UEs with pending data were not being considered during RC event generation, ensuring proper scheduling priority for active UEs.
