## Title: Fix signed/unsigned integer comparison mismatch in Msg4 retransmission check

### Summary
The `check_Msg4_retransmission` function in the eNB MAC scheduler contained a signed/unsigned integer comparison mismatch that could lead to incorrect branching behavior and compiler warnings. The local variable `round` was declared as a signed `int` but was being compared against unsigned values in the HARQ status checking logic, creating a type mismatch that violates C language semantics and could cause subtle bugs in boundary conditions.

This patch corrects the type declaration by changing `round` from `int` to `uint8_t`, matching the actual usage context and the unsigned nature of HARQ round numbers. This eliminates the signed/unsigned comparison warning and ensures correct behavior when checking retransmission status.

### Changes
- `openair2/LAYER2/MAC/eNB_scheduler_RA.c`: Changed the type of local variable `round` from `int` to `uint8_t` in the `check_Msg4_retransmission` function (line 1129). This aligns the variable type with its usage in HARQ round comparisons and eliminates signed/unsigned mismatch warnings.

### Implementation Details
The `round` variable is used to track HARQ retransmission round numbers, which are inherently non-negative and typically bounded to a small range (0-7 for most HARQ configurations). Using `uint8_t` is more appropriate than `int` as it:
- Correctly represents the unsigned nature of round counters
- Matches the type used in related HARQ data structures
- Eliminates potential sign extension issues during comparisons
- Reduces memory footprint (though minimal in this case)
