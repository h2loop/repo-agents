## Title: Fix heap buffer overflow in SRS beamforming report packing

### Summary
The `pack_nr_srs_beamforming_report` function in the nFAPI layer is vulnerable to a heap buffer overflow when packing SRS beamforming reports. The function iterates over `num_reported_symbols` and `num_prgs` fields from the incoming message without validating these values against reasonable bounds. Malformed or corrupted messages with excessively large values could cause the packing logic to write far beyond the allocated buffer, leading to memory corruption and potential security vulnerabilities.

This patch adds explicit bounds checking for both `num_reported_symbols` and `num_prgs` before entering their respective packing loops. If either value exceeds the maximum defined by 3GPP specifications (4 for reported symbols, 272 for PRGs), the function now returns 0 to indicate failure, preventing the buffer overflow.

### Changes
- `nfapi/open-nFAPI/nfapi/src/nfapi_p7.c`: 
  - Added validation in `pack_nr_srs_reported_symbol()` to check `num_prgs <= 272` before processing
  - Added validation in `pack_nr_srs_beamforming_report()` to check `num_reported_symbols <= 4` before the packing loop
  - Both functions return 0 on validation failure to maintain consistent error handling

### Implementation Details
The validation limits are derived from 3GPP TS 38.331 and TS 38.214 specifications:
- `num_reported_symbols` is limited to 4 (maximum SRS symbols per slot)
- `num_prgs` is limited to 272 (maximum number of PRBs in NR)

The checks are placed at function entry to fail fast and avoid unnecessary processing. The return value of 0 aligns with existing error handling patterns in the nFAPI packing functions.
