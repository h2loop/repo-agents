## Title: Fix NAS message integrity check bypass logic to prevent false positives

### Summary
The NAS message integrity checking code contained an overly broad bypass mechanism (`NAS_CODE_TO_BE_MODIFIED`) that forced MAC validation to pass under certain conditions, causing false positives and missed error detection in the network simulator. This patch replaces the hardcoded bypass with a targeted approach that only allows integrity check bypass for specific initial NAS messages that are legitimately sent without security protection during UE registration.

### What Changed
The MAC validation logic now properly distinguishes between:
- **Legitimate unprotected messages**: Initial NAS messages (ATTACH_REQUEST, IDENTITY_RESPONSE, AUTHENTICATION_RESPONSE, SECURITY_MODE_COMPLETE) that can be sent before a security context is established
- **Messages requiring protection**: All other NAS messages that must have valid MAC integrity protection

### Affected Files
- `openair3/NAS/COMMON/API/NETWORK/nas_message.c`: Refactored the integrity check bypass logic in `nas_message_decode()` to decode message type first and only bypass MAC validation for explicitly allowed initial NAS message types when no security context exists.

### Implementation Details
The fix introduces message type detection before applying the bypass:
1. Checks if the protocol discriminator is EPS_MOBILITY_MANAGEMENT_MESSAGE
2. Verifies the message type against a whitelist of unprotected initial messages
3. Only bypasses MAC check when: no security context exists, computed MAC is 0, message MAC is non-zero, AND message type is in the whitelist
4. Returns `TLV_DECODE_MAC_MISMATCH` for all other MAC validation failures

This ensures proper security validation while maintaining compatibility with 3GPP-specified initial registration flows that legitimately operate without integrity protection.
