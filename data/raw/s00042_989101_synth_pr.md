## Title: Fix ITTI scheduling priority inversion

### Summary
The ITTI (Inter-Task Interface) message handling suffers from a priority inversion issue where non-message file descriptor events can be processed before higher-priority message queue events. The root cause is in `itti_receive_msg()`, which uses a legacy loop condition that exits when multiple file descriptors are present, even if the message queue is empty. This allows lower-priority subscribed FD events to be serviced before messages, violating the intended priority scheme where task-to-task messages should always take precedence.

The fix ensures strict priority ordering by:
1. Processing message queue events (sem_fd) before any other subscribed FD events in `itti_get_events_locked()`
2. Modifying `itti_receive_msg()` to block until at least one message is available, regardless of other FD activity
3. Removing the legacy behavior that allowed returning NULL messages when other FD events were present

### Changes
- `common/utils/ocp_itti/intertask_interface.cpp`:
  - In `itti_get_events_locked()`: Split event processing into two passes—first for message queue events, then for other FD events—to guarantee message priority
  - In `itti_receive_msg()`: Replaced the problematic `do-while` loop with a clear `while` loop that waits exclusively for messages; removed NULL message return path
  - Added comments documenting the priority scheme

### Implementation Details
The message queue eventfd (`sem_fd`) now gets priority treatment by being processed in a separate initial pass. The receive function no longer considers other FD events as a condition for proceeding, ensuring that tasks always consume pending messages before handling lower-priority I/O events. This eliminates the race condition where epoll could signal both message and non-message events, but the legacy logic would allow processing non-message events first.
