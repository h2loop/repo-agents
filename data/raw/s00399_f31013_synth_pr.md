## Title: Fix PDCP initialization order to resolve configuration mismatch

### Summary
Reorder initialization calls in `nr_pdcp_layer_init()` to ensure the PDCP data indication queue is established before E1 interface initialization. The previous order caused a configuration mismatch where the E1 interface initialization could attempt to configure or access PDCP data structures before they were properly set up, leading to inconsistent behavior in split gNB architectures (CU-CP/CU-UP).

The root cause was a dependency violation: `nr_pdcp_e1_if_init()` relies on the PDCP data indication queue being ready, but was being called before `init_nr_pdcp_data_ind_queue()`. This race condition particularly impacted scenarios where E1 interface configuration needed to register callbacks or configure queue parameters during initialization. The fix ensures proper layer initialization sequence by establishing the data path infrastructure before protocol interface setup.

### Changes
- `openair2/LAYER2/nr_pdcp/nr_pdcp_oai_api.c`: Swapped the order of `nr_pdcp_e1_if_init()` and `init_nr_pdcp_data_ind_queue()` calls in `nr_pdcp_layer_init()` to guarantee the PDCP data indication queue is initialized first.

### Implementation Details
The change is a simple reordering of two adjacent function calls. The PDCP data indication queue initialization must complete before E1 interface setup because the E1 interface may depend on queue resources being available during its configuration phase. This ensures proper synchronization between protocol layers during gNB startup.

### Testing
- Verified initialization sequence in both monolithic gNB and split CU-CP/CU-UP architectures
- Confirmed PDCP data path establishment completes without configuration errors
- Ran basic connectivity tests to ensure no regression in data indication handling
