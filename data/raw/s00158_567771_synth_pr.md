## Title: Add UE state validation before protocol message processing in LTE UE sync thread

### Summary
The LTE UE synchronization thread previously proceeded to process protocol messages without validating the UE's L2 connection state. When operating in `no_L2_connect` mode, this unnecessary downstream processing could consume resources and potentially trigger undefined behavior. This patch adds an explicit state validation check immediately after thread initialization to short-circuit the thread when L2 connectivity is disabled.

The validation ensures that UE instances configured for PHY-only operation (without MAC/RRC layers) skip protocol message processing entirely, improving efficiency and preventing potential issues in stub/simulation modes.

### Changes
- `executables/lte-ue.c`: Added UE mode validation in `UE_thread_synch()` after `init_thread()` initialization. If `UE->mode == no_L2_connect`, the function now logs a warning and returns immediately before entering the main synchronization loop.

### Implementation Details
- The check is positioned immediately following the `init_thread()` call at line 433, ensuring thread priority setup completes before validation
- Uses the existing `UE->mode` enumeration to detect `no_L2_connect` state
- Logs a warning-level message identifying the affected UE instance for debugging visibility
- Returns the thread's exit value pointer directly, avoiding any protocol message processing or synchronization attempts
- Maintains backward compatibility by allowing normal execution for all other UE modes

### Testing
- Verified UE behavior in `no_L2_connect` mode: thread exits cleanly without protocol processing
- Confirmed normal operation unaffected in standalone and PNF modes via `lte-ue-rfsim` test suite
- Validated warning message appears in logs when running with `no_L2_connect` configuration
