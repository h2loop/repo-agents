## Title: Fix incorrect field lookup logic in T tracer get_field function

### Summary
The `get_field` function in the T tracer utility contains a logic error that causes it to return incorrect field indices when searching database event formats. When a field with matching name but mismatched type is encountered, the function breaks out of the loop but still returns the index of the mismatched field, rather than continuing the search.

This incorrect field indexing propagates downstream to MCS selection algorithms, causing them to retrieve wrong parameters from trace events. The result is MCS values that don't match actual channel quality or UE capability, particularly problematic in multi-RRU scenarios with complex trace events.

The bug was introduced by incorrect conditional logic that used `break` instead of continuing the search when a type mismatch occurred. This fix corrects the logic to only return a field index when both name and type match.

### Changes
- `common/utils/T/tracer/hacks/multi-rru-clean.c`: Fixed the field type matching logic in `get_field()` to properly continue searching when a name matches but type doesn't match. The function now only returns an index when both conditions are satisfied.

### Implementation Details
The fix changes the nested conditional from:
```c
if (strcmp(f->type[i], type)) break;
return i;
```

To:
```c
if (!strcmp(f->type[i], type))
  return i;
```

This ensures proper field resolution by continuing the search loop until an exact match is found, rather than prematurely returning an incorrectly typed field.
