## Title: Add critical error logging to M2AP ERROR_INDICATION handler

### Summary
The `eNB_handle_ERROR_INDICATION` function in the M2AP module is called when a critical ERROR_INDICATION message is received from the MCE (Multi-cell/Multicast Coordination Entity). Previously, this function would immediately trigger an `AssertFatal` without logging any context about the error, making it impossible to debug which connection or stream triggered the failure in production environments. This missing log statement hindered root cause analysis of downstream issues.

This fix adds a critical error log statement that captures the `assoc_id` and `stream` parameters before the assertion fires. This provides essential debugging information about the SCTP association and stream that delivered the error indication, enabling operators to trace the source of M2AP protocol errors without requiring core dumps or crash analysis.

### Changes
- `openair2/M2AP/m2ap_eNB_interface_management.c`: Added `LOG_E` statement at the entry of `eNB_handle_ERROR_INDICATION` to log the `assoc_id` and `stream` parameters before the not-implemented assertion.

### Implementation Details
The log message uses the `LOG_E` macro (error level) with the M2AP module identifier to ensure it appears in error logs. It includes the function name and both key parameters (`assoc_id` and `stream`) to provide complete context about the failing connection. The log is placed immediately at function entry to capture state before any future implementation changes modify the control flow.

### Testing
- Verified the change compiles successfully within the OAI 5G build system
- Confirmed the log statement follows OAI logging conventions used elsewhere in the M2AP module
- The function currently terminates with `AssertFatal`, so the log will appear in crash logs when this unimplemented code path is triggered
