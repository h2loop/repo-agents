## Title: Fix HARQ redundancy version handling in MCH PDU processing

### Summary
Fixed a HARQ process configuration error in the multicast channel (MCH) PDU processing path. The `handle_nfapi_mch_pdu` function was incorrectly hardcoding the redundancy version (`rvidx`) to 0 instead of using the actual value provided in the incoming PDU's REL-8 structure. This caused improper HARQ combining behavior for multicast transmissions, leading to potential data corruption and throughput degradation. The fix ensures the HARQ process uses the correct redundancy version as specified by the upper layers.

### Changes
- **openair1/SCHED/fapi_l1.c**: Line 166 in `handle_nfapi_mch_pdu()` changed from `dlsch->harq_processes[0]->rvidx = 0;//rvidx;` to `dlsch->harq_processes[0]->rvidx = rel8->redundancy_version;`. This assigns the proper redundancy version from the `rel8` structure instead of using a hardcoded value.

### Implementation Details
The root cause was identified downstream of the initially suspected M2AP subsystem. The HARQ initialization error actually originated in the PHY layer scheduler's MCH PDU handler. The commented-out variable `rvidx` suggests this was a known placeholder that was never properly wired to the actual PDU data. The fix directly accesses the `redundancy_version` field from the `rel8` structure, which contains the correctly parsed PDU parameters.

### Testing
- Verified MCH PDU processing correctly propagates redundancy version from NFAPI messages
- Confirmed HARQ buffer management behaves correctly across all RV indices (0-3)
- Validated with multi-cell multicast simulations showing improved BLER performance
