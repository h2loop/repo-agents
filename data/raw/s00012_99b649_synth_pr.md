## Title: Fix deadlock in VNF simulator MAC by adopting RAII lock guards

### Summary
The VNF simulator MAC implementation suffers from a deadlock caused by manual mutex lock/unlock management in the `mac_private` class. The `allocate_mac_pdu()`, `release_mac_pdu()`, `push_rx_buffer()`, and `pop_rx_buffer()` methods use explicit `mutex.lock()` and `mutex.unlock()` calls, creating vulnerability to deadlocks from inconsistent lock ordering and missing unlocks on exception paths. This patch eliminates the deadlock risk by replacing all manual mutex operations with C++ RAII-style `std::lock_guard<std::mutex>`.

### Changes
- **nfapi/open-nFAPI/vnf_sim/src/mac.cpp**: Replace manual mutex management with `std::lock_guard<std::mutex>` in four critical methods:
  - `allocate_mac_pdu()`: Lock guard ensures mutex release after PDU allocation from free store
  - `release_mac_pdu()`: Lock guard protects free store push operation
  - `push_rx_buffer()`: Lock guard secures RX buffer queue insertion
  - `pop_rx_buffer()`: Lock guard guarantees unlock on both success and empty buffer paths

### Implementation Details
The `std::lock_guard` automatically releases the mutex when the function scope exits, providing exception safety and preventing deadlocks that could occur if exceptions were thrown between manual lock/unlock calls. This change maintains the same locking semantics while making the code robust against deadlock conditions. The transformation is mechanical and preserves all existing thread-safety guarantees.

### Testing
- Verified deadlock no longer reproduces under concurrent access patterns that previously triggered the issue
- Confirmed thread safety is maintained for all four modified methods under high contention scenarios
- Validated that the VNF simulator runs without regression in standard integration tests
