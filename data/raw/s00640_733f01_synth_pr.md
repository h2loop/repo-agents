## Title: Add error handling for E1 bearer modification requests

### Summary
Fix an unchecked return value issue in the E1 bearer update mechanism that could lead to silent failures when sending bearer modification requests to the CU-UP. The `e1_send_bearer_updates()` function previously returned void and would attempt to send messages with invalid association IDs, causing potential memory leaks and lost bearer updates during UE context modification procedures.

The function now validates the CU-UP association ID before sending the request and returns an error code to callers. This prevents sending messages over invalid SCTP associations and ensures proper error propagation when CU-UP connectivity is not available.

### Changes
- `openair2/RRC/NR/rrc_gNB.c`:
  - Modified `e1_send_bearer_updates()` to return `int` (0 on success, -1 on failure) instead of `void`
  - Added validation check for `assoc_id <= 0` before calling `bearer_context_mod()`
  - Added proper cleanup of `req.secInfo` on error paths
  - Updated call site in `rrc_CU_process_ue_context_modification_response()` to check return value and log errors

### Implementation Details
The fix adds defensive programming at the E1 interface layer:
- Early exit with error if CU-UP association ID is invalid
- Memory leak prevention by freeing security info structure on failure
- Error logging to aid debugging of CU-UP connectivity issues
- Maintains backward compatibility by preserving original success behavior
