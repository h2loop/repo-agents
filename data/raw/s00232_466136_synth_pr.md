## Title: Fix missed scheduling opportunity in ESM Status message handler

### Summary
The ESM Status message handler in the UE NAS stack was not triggering scheduling for pending data after processing status messages. When the UE had pending data to send on active bearers following an ESM Status exchange, the lower layers were not notified, causing a missed scheduling opportunity and potential data transmission delays.

This patch modifies the `esm_recv_status()` function to accept a `nas_user_t` parameter and checks for pending data on active bearers after successful status processing. If pending data is detected, the function sets the scheduling request flag (`sched_req`) to ensure the UE is properly scheduled for uplink transmission. This ensures that data transmission is not stalled after ESM status procedures.

### Changes
- `openair3/NAS/UE/ESM/SAP/esm_recv.c`: Modified `esm_recv_status()` to accept `nas_user_t *user` parameter and added logic to check for pending data on active bearers, setting `sched_req` flag when needed
- `openair3/NAS/UE/ESM/SAP/esm_recv.h`: Updated function prototype for `esm_recv_status()` to include `nas_user_t *user` parameter
- `openair3/NAS/UE/ESM/SAP/esm_sap.c`: Updated call site to pass the `user` parameter when calling `esm_recv_status()`

### Testing
- Verified that UE with pending data is properly scheduled after ESM Status message exchange
- Tested scenarios with multiple active bearers to ensure correct scheduling request generation
- Confirmed no regression in existing ESM status handling procedures
