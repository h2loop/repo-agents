## Title: Fix uninitialized memory reads in user API notification handlers

### Summary
Fix three instances where incorrect pointer values were passed to `user_ind_notify()`, causing the notification handlers to read beyond the bounds of valid data and access uninitialized memory. The `user_api_emm_callback()` and `user_api_esm_callback()` functions were passing pointers to the entire `user_ind_t` structure instead of pointers to the specific union members (`reg`, `loc`, or `pdn`) expected by the respective handlers. This resulted in the `_user_api_registration_handler` and related handlers reading past the intended struct boundaries into uninitialized memory when casting the `void*` data parameter.

### Changes
- `openair3/NAS/UE/API/USER/user_api.c`: 
  - Line 504: Changed `(void*)&ind` to `(void*)&ind.notification.reg` for `USER_IND_REG` notification
  - Line 514: Changed `(void*)&ind` to `(void*)&ind.notification.loc` for `USER_IND_LOC` notification  
  - Line 552: Changed `(void*)&ind` to `(void*)&ind.notification.pdn` for `USER_IND_PDN` notification

### Implementation Details
The `user_ind_t` structure contains a union where only one member (`notification.reg`, `notification.loc`, or `notification.pdn`) is actively used per notification type. The handlers expect pointers to these specific union members, not the enclosing structure. Passing `&ind` caused the handlers to read beyond the size of the intended member type, triggering undefined behavior. The fix passes the correct member pointer, ensuring handlers only access properly initialized memory.

### Testing
- Verified compilation succeeds with the updated pointer arithmetic
- No functional behavior changes; this is a memory safety bug fix only
