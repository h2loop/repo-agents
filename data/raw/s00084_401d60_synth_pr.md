## Title: Add NULL checks after connection info lookup in UL node sync handlers

### Summary
The UL node sync handlers in the nFAPI VNF component dereference pointers returned by `vnf_p7_connection_info_list_find()` without checking for NULL, leading to potential segmentation faults when processing messages from unknown PHY instances. If a UL_NODE_SYNC message arrives with a phy_id that doesn't exist in the connection list, the function returns NULL, and subsequent accesses to `phy->sfn_sf` (LTE) or `phy->sfn/phy->slot` (NR) trigger undefined behavior.

This defensive fix adds NULL checks in both LTE and NR UL node sync processing paths. When an unknown phy_id is encountered, the handlers now log an error message and return early instead of crashing. This improves robustness against misconfiguration, race conditions during connection teardown, or malformed messages.

### Changes
- `nfapi/open-nFAPI/vnf/src/vnf_p7.c`: Added NULL checks after `vnf_p7_connection_info_list_find()` calls in:
  - `vnf_handle_ul_node_sync()` (line 1107): Returns early if phy connection info is NULL
  - `vnf_nr_handle_ul_node_sync()` (line 1643): Returns early if phy connection info is NULL

### Implementation Details
Both handlers now follow the same pattern: after retrieving the connection info pointer, they check for NULL and log an error with the unknown phy_id before returning. This prevents the NULL pointer dereference while providing diagnostic information for debugging connection mismatches.

### Testing
This is a defensive programming fix that handles edge cases not covered by existing tests. The change is minimal and follows established error handling patterns elsewhere in the codebase.
