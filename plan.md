# Plan: Generate SFT Training Data for OpenAirInterface5G Using SERA/SVG Pipeline

## Scope

This plan covers **data generation only** — from function extraction through to producing a filtered, formatted SFT-ready dataset. Training, evaluation, and deployment are out of scope and will be planned separately.

**No local system-level changes will be made.** All Docker builds, package installs, and environment modifications happen inside containers or virtual environments only.

---

## Context

Adapt the SERA paper's SVG (Soft Verified Generation) pipeline to generate SFT training data from the OpenAirInterface5G (OAI5G) codebase. OAI5G is a large C/C++ 5G telecommunications codebase implementing the full 4G/5G protocol stack (PHY, MAC, RLC, PDCP, RRC, NAS). SERA was only tested on Python repos, so this requires adaptation for C/C++ and telecom-domain specifics.

**Teacher model**: Qwen3-Coder-480B via LiteLLM proxy on GCP (`qwen/qwen3-coder-480b-a35b-instruct-maas`)
**LiteLLM endpoint**: `https://litellm-prod-909645453767.asia-south1.run.app/v1`
**Compute**: Cloud GPUs on-demand (for Docker containers running SWE-agent rollouts)
**Scope**: Full OAI5G repository
**Target output**: ~8,000-16,000 filtered SFT trajectories in JSONL format

## Phase 1: C/C++ Function Extraction

Extract all functions from the OAI5G codebase to serve as random starting points for SVG rollouts.

### Implementation

**Create `scripts/extract_functions.py`** — uses `tree-sitter` with C and C++ grammars to parse all `.c`, `.cpp`, `.h` files and extract:
- Function name
- File path (relative to repo root) and line range
- Function signature (return type, parameters)
- Approximate token count of function body
- Parent directory / subsystem tag (e.g., `openair1/PHY`, `openair2/LAYER2/MAC`)

```
Approach:
1. Use tree-sitter-c and tree-sitter-cpp (installed in project venv, NOT system-level)
2. Walk .c/.cpp/.h files under openair1/, openair2/, openair3/, common/, executables/, radio/, nfapi/
3. Parse each file, extract function_definition nodes
4. Store as JSONL: {name, file, start_line, end_line, signature, subsystem, token_estimate}
```

**Filtering criteria** (exclude from function pool):
- Functions shorter than 5 lines (trivial getters/setters)
- Test functions (in `tests/` directory)
- Generated/auto-generated code (check for generation markers like `/* Generated by */`)
- Static inline functions in headers (typically trivial)

**Expected output**: ~15,000-30,000 extractable functions.

### Deliverables
- `scripts/extract_functions.py`
- `data/oai5g_functions.jsonl`

## Phase 2: Bug Prompt Taxonomy for C/C++ Telecom Code

SERA uses 51 bug types. We create an expanded taxonomy covering C/C++ bugs AND telecom/5G domain bugs.

### Bug Categories

**A. Generic C/C++ bugs (~40 types):**

| Category | Bug Types |
|----------|-----------|
| Memory management | buffer overflow, use-after-free, double free, memory leak, null pointer dereference, uninitialized memory read |
| Concurrency | race condition, deadlock, missing mutex lock/unlock, thread-unsafe shared state access |
| Integer issues | integer overflow/underflow, signed/unsigned mismatch, truncation on assignment |
| Control flow | off-by-one, missing break in switch, unreachable code, incorrect loop bounds, missing return |
| Type safety | incorrect cast, type confusion, implicit narrowing conversion |
| Error handling | unchecked return value, missing error propagation, incorrect errno usage, missing NULL check |
| Resource management | file descriptor leak, socket not closed, missing cleanup on error path |
| API misuse | incorrect argument order, wrong flag combination, deprecated API call |
| String handling | buffer overrun in string ops, format string mismatch, missing null terminator |
| Code quality | dead code, redundant computation, magic number, copy-paste error |

**B. Telecom/5G domain-specific bugs (~25 types):**

| Category | Bug Types |
|----------|-----------|
| Protocol state machine | incorrect state transition, missing state validation, stale state on re-entry |
| Timer management | incorrect timer duration, missing timer start/stop, timer expiry race |
| Message encoding/decoding | ASN.1 encoding error, incorrect IE handling, missing optional field check |
| Configuration | parameter range check missing, default value wrong, config mismatch between layers |
| Scheduling | incorrect resource block allocation, missed scheduling opportunity, priority inversion |
| Channel processing | incorrect HARQ process handling, wrong MCS selection, CRC computation error |
| Signaling | missing mandatory IE in message, incorrect procedure sequence, duplicate message handling |
| Radio resource | incorrect power control calculation, beam management error, frequency allocation overlap |
| Interface handling | incorrect F1AP/E1AP/NGAP message construction, cross-layer interface mismatch |
| Logging/telemetry | missing critical log, incorrect log level, performance-impacting debug left enabled |

### Prompt Template

```
There is a {bug_description} downstream of function {func_name} in the {subsystem} subsystem.
The function is located at {file_path}:{start_line}.
Please investigate and fix this issue.
```

### Deliverables
- `configs/bug_prompts.json` — structured list of ~65 bug types with `bug_id`, `description`, `domain`, `subsystems`
- `configs/bug_prompt_template.txt`

## Phase 3: Docker Environment Setup

OAI5G needs a Docker container for SWE-agent rollouts. **All environment setup happens inside Docker — no local system changes.**

### Implementation

**Create `docker/Dockerfile.sera`**:
```dockerfile
FROM ubuntu:22.04

# System deps for code navigation (NOT full OAI build)
RUN apt-get update && apt-get install -y \
    gcc g++ cmake make \
    clang-format cscope universal-ctags \
    git python3 python3-pip \
    grep findutils coreutils \
    && rm -rf /var/lib/apt/lists/*

# Copy OAI5G source into container
COPY openairinterface5g/ /repo/

WORKDIR /repo
```

Key design decisions:
- Container provides source code + basic C/C++ tooling for navigation
- Does NOT attempt full OAI5G compilation (avoids UHD, DPDK, hardware SDK deps)
- Agent can use `gcc -fsyntax-only` for lightweight syntax checks
- Working directory is `/repo` (SWE-agent convention)

### Multi-Snapshot Strategy

Generate data from 5 equally-spaced commits spanning the last 2 years:
1. Select 5 commit SHAs from OAI5G git history
2. Build a Docker image per commit
3. Ensures trajectory diversity across codebase evolution

### Deliverables
- `docker/Dockerfile.sera`
- `docker/build_sera_container.sh`
- `configs/commits.json` — list of 5 commit SHAs

## Phase 5: SVG Pipeline — Data Generation Scripts

The core pipeline producing raw trajectories.

### 5.1 Rollout 1: Change Generation

**`scripts/rollout1.py`**:
1. Sample random function from `data/oai5g_functions.jsonl`
2. Sample random bug type from `configs/bug_prompts.json`
3. Format prompt using template
4. Launch SWE-agent against Docker container with the prompt
5. Teacher model (Qwen3-Coder-480B) drives the agent
6. Collect trajectory T1 and extract patch P1 via `git diff`
7. Self-evaluation: ask teacher if the change was aligned with the prompt
8. If rejected (teacher says change was unrelated), retry up to 3 times with different bug type
9. Save T1 (full trajectory JSONL) and P1 (unified diff)

### 5.2 Synthetic PR Generation

**`scripts/generate_pr.py`**:
1. Load T1 trajectory
2. Load a demonstration PR from `configs/demo_prs/` (curated from real OAI5G merge requests)
3. Prompt teacher: "Given the following agent trajectory, write a pull request description following this format: {demo_PR}"
4. Save synthetic PR description

**Curate 10-20 demonstration PRs** from OAI5G's GitLab history covering:
- Bug fixes, refactoring, feature additions, protocol compliance fixes

### 5.3 Rollout 2: Reproduction

**`scripts/rollout2.py`**:
1. Reset Docker container to clean state (same commit as rollout 1)
2. Provide SWE-agent with ONLY the synthetic PR description
3. Teacher attempts to reproduce the change
4. Collect T2 and P2

### 5.4 Soft Verification

**`scripts/soft_verify.py`**:
1. Parse P1 and P2 into sets of changed lines (added/modified)
2. Compute line-level recall: `r = |P2 ∩ P1| / |P1|`
3. Output verification score and classification:
   - `r = 1.0` → hard-verified
   - `0.5 <= r < 1.0` → soft-verified
   - `0 < r < 0.5` → weakly verified
   - `r = 0` → unverified

Per SERA's findings: verification threshold has minimal effect on downstream SFT performance. We collect all trajectories but track scores for analysis.

### 5.5 Master Orchestrator

**`scripts/generate_data.py`**:
```
For each commit_snapshot in commits.json:
  Spin up Docker container for this commit
  Load functions extracted for this commit
  For batch of functions (parallelized):
    1. rollout1(func_i, random_bug_j) → T1, P1
    2. generate_pr(T1) → synth_PR
    3. rollout2(synth_PR) → T2, P2
    4. soft_verify(P1, P2) → score
    5. Save all artifacts to data/raw/
```

**Parallelization**: Pool of Docker containers, async rollout dispatch. Respect LiteLLM rate limits. Target: 100-500 trajectories/day depending on API throughput.

### Deliverables
- `scripts/rollout1.py`
- `scripts/generate_pr.py`
- `scripts/rollout2.py`
- `scripts/soft_verify.py`
- `scripts/generate_data.py`
- `configs/demo_prs/` — 10-20 curated demonstration PRs

## Phase 6: Data Filtering and Quality Control

Filter raw trajectories into a clean SFT-ready dataset.

### Filtering Pipeline

**`scripts/filter_data.py`**:

| Filter | Criterion | Rationale (SERA reference) |
|--------|-----------|---------------------------|
| Self-evaluation | Exclude rejected trajectories | ~2% rejection rate expected (SERA Section 3) |
| Patch size | Patch ≤ 40 lines | Improves specialization quality (SERA Table 5) |
| Tool output length | Avg tool output ≤ 600 tokens | Reduces noise (SERA Table 5) |
| Duplicate patches | Remove exact-duplicate patches | Prevent memorization |
| Truncation ratio | Ratio ≥ 0.88 | Best SFT performance at this threshold (SERA Section 5.2) |
| Token length | Prioritize trajectories ≤ 32768 tokens | Fits training context window |

### Target Dataset Composition

- **5,000 filtered T1 trajectories** (diverse, from rollout 1)
- **3,000 soft-verified T2 trajectories** (PR-driven, from rollout 2, r ≥ 0.5)
- **Total: 8,000 trajectories** for initial specialization

To produce 8,000 filtered trajectories, expect to generate ~16,000-20,000 raw trajectories (assuming ~40-50% pass all filters).

### Deliverables
- `scripts/filter_data.py`
- `data/filtered/` — filtered trajectory set with statistics report

## Phase 7: Format SFT Dataset

Convert filtered trajectories into the final SFT training format.

### Implementation

**`scripts/format_for_training.py`**:

Convert each trajectory into a multi-turn conversation in JSONL:
```json
{
  "conversations": [
    {"role": "system", "content": "<SWE-agent system prompt with tool definitions>"},
    {"role": "user", "content": "<task description / PR description>"},
    {"role": "assistant", "content": "<reasoning + tool call>"},
    {"role": "tool", "content": "<tool observation>"},
    {"role": "assistant", "content": "<reasoning + next tool call>"},
    ...
  ],
  "metadata": {
    "source": "oai5g_sera",
    "rollout_type": "T1|T2",
    "verification_score": 0.85,
    "commit": "abc123",
    "subsystem": "openair2/LAYER2/MAC",
    "bug_type": "race_condition",
    "token_count": 28450,
    "truncation_ratio": 0.95
  }
}
```

Key formatting rules (from SERA Section 7 — deployment lessons):
- Tool format must exactly match SWE-agent conventions (whitespace, directory listing format, etc.)
- Include reasoning traces from teacher model (critical — SERA Table 8 shows 18% performance drop without them)
- Preserve the full agent loop: reasoning → tool call → observation → reasoning → ...

### Deliverables
- `scripts/format_for_training.py`
- `data/sft_dataset/oai5g_train.jsonl` — final SFT-ready dataset
- `data/sft_dataset/oai5g_held_out.jsonl` — 10% held-out for future evaluation
- `data/sft_dataset/dataset_stats.json` — summary statistics

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Qwen3-Coder-480B may struggle with complex telecom C code | Start pilot with simpler subsystems (`common/`). Inspect trajectory quality early. Consider Kimi-K2 or DeepSeek as fallback teachers. |
| C/C++ functions much larger than Python → longer trajectories | Use aggressive truncation ratio filtering (≥ 0.88). Bias function sampling toward smaller functions initially. |
| Telecom domain bugs too specialized for teacher model | Weight generic C/C++ bugs at 70%, telecom bugs at 30% initially. Adjust after pilot. |
| LiteLLM proxy rate limits slow generation | Measure throughput during pilot. Scale parallelism to match. Budget for multi-day generation. |
| Docker container missing needed tools | Start minimal, add tools only if agent rollouts show navigation failures during pilot. |
| Patch quality too low (trivial or nonsensical changes) | Self-evaluation filter catches ~2%. Add manual review of pilot data. Iterate on prompt template if needed. |
